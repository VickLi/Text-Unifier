## 三、性能优化记录

### 3.1 后端性能优化

| 优化项 | 措施 | 预期收益 |
| :--- | :--- | :--- |
| **并行文件处理** | 使用 `rayon::par_iter()` 并行读取和归一化多个文件 | 多文件场景提升 3-5 倍速度 |
| **SHA-256 指纹** | 使用哈希值作为 Map Key，O(1) 查找复杂度 | 10MB 文本处理 < 100ms |
| **二次校验机制** | Hash 匹配后进行字符串 `eq` 比较 | 杜绝哈希碰撞导致的误判 |
| **写入原子性** | 先写临时文件再 rename | 防止导出过程中断导致文件损坏 |
| **Release 优化** | `strip = true`, `lto = true`, `codegen-units = 1` | 减小二进制体积约 30%，提升运行时性能 |

### 3.2 前端性能优化

| 优化项 | 措施 | 预期收益 |
| :--- | :--- | :--- |
| **组件 memo** | `DuplicateItem` 和 `PreviewParagraph` 使用 `React.memo` | 减少不必要的重渲染 |
| **纯前端过滤** | 勾选删除仅在前端内存中计算，无需调用后端 | 预览刷新延迟 < 16ms (60fps) |
| **Zustand 订阅** | 精细化的 store 订阅（每个组件只订阅所需字段） | 避免无关状态变化触发重渲染 |
| **Zustand 替代全局状态** | 使用 Zustand 替代 React Context | 避免 Context 导致的整树重渲染 |

### 3.3 内存优化

| 优化项 | 措施 | 预期收益 |
| :--- | :--- | :--- |
| **段落内容单份存储** | 后端 HashMap 中每段内容仅存一份 | 内存峰值约为原始文本大小的 1.2 倍 |
| **预览段落存指针** | 预览段落只存储内容哈希，不复制文本 | 降低内存占用 |
| **Set 数据结构** | 使用 `Set<string>` 而非数组存储勾选状态 | O(1) 查找，勾选/取消操作极快 |
