## 一、Bug 修复概要

### BUG-V3.2.3-001 (P0): 全窗口拖拽文件添加仍然无效

**问题描述**：  
V3.2.2 版本添加了 `dragover` 事件处理（`preventDefault()`），理论上修复了 HTML5 Drag and Drop 规范中 `drop` 事件必须由 `dragover` 的 `preventDefault()` 才能触发的条件。但用户实测反馈：从 Windows 资源管理器拖拽 .txt 文件到应用窗口 → 蓝色遮罩正常显示"释放到此添加 TXT 文件" → 释放鼠标 → 文件不会被加载（文件标签栏无新增文件，无任何错误提示）。

**根本原因分析**：

经过对完整事件链、@dnd-kit/core 源码、Electron 31 特性的深入排查，定位到以下两个根因：

#### 根因 1：`File.path` 异步填充导致的取值为空（主因）

在 Electron 31 中，拖放事件（DragEvent）的 `File.path` 属性虽然仍存在，但其填充时机存在竞态条件：
- **文件对话框**（`<input type="file">`）场景：用户点击 → 文件对话框打开 → 用户选择 → 对话框关闭 → `onChange` 触发。此时 `File.path` 已由 Electron 内部异步填充完成，取值正常。
- **拖放**场景：用户释放鼠标 → `drop` 事件立即同步触发 → 此时 `File.path` 可能尚未完成异步填充，取值为 `undefined` 或空字符串 `""`。

代码 `(f as any).path || f.name` 在 `File.path` 为空时退化为仅返回文件名（如 `"test.txt"`），无完整路径。后续 `scanFiles(["test.txt"])` 调用 `fs.existsSync("test.txt")` 因当前工作目录不匹配而返回 `false`，导致主进程抛出 `"没有有效的 .txt 文件"` 错误。

**修复方案**：  
使用 Electron 28+ 推荐的 `webUtils.getPathForFile(file)` API，该 API **同步且可靠**地返回 `File` 对象的文件系统路径，不依赖异步填充。三层降级策略：
1. **`webUtils.getPathForFile(file)`**（首选）— Electron 28+ 原生 API
2. **`(file as any).path`**（降级）— 传统 `File.path`，向后兼容
3. **`file.name`**（兜底）— 仅文件名

#### 根因 2：缺少 `will-navigate` 事件阻止（次因）

Electron 的默认行为中，文件拖放到 `BrowserWindow` 时可能触发页面导航（尝试打开 `file://` 协议）。即使导航被安全策略阻止，该行为可能消耗 `drop` 事件，导致渲染进程的 JavaScript 监听器无法接收到 `drop` 事件。

**修复方案**：  
在 `electron/main.ts` 的 `createWindow()` 中添加 `webContents.on('will-navigate')` 监听，显式阻止所有 `file://` 协议的导航尝试。

#### 额外加固：`dataTransfer.files` 访问的 try-catch 保护

在 `contextIsolation: true` 的渲染进程中，访问 `e.dataTransfer.files` 可能因跨域安全策略抛出异常。添加 try-catch 包装，防止未捕获异常导致静默失败。

## 六、部署后发现的修复

> 以下修复在实际部署验证中发现，已合并到 v3.2.3 中

### BUG-V3.2.3-003 (P0): asar 中 native 模块路径解析失败

**问题描述**：  
编译后的 `dist-electron/main.cjs` 使用 `path.join(__dirname, '../native/index.js')` 加载原生模块。当应用打包为 `app.asar` 时，`__dirname` 指向 asar 虚拟文件系统内部路径（`resources/app.asar/dist-electron/`），`../native/` 被解析为 `resources/app.asar/native/`（仍在 asar 内部），但 `native/` 目录并未打包进 asar，导致模块找不到。

**修复方案**：
1. 使用 `process.resourcesPath` 定位到真实的 resources 目录（不受 asar 影响）
2. 构建路径为 `path.join(process.resourcesPath, 'native/index.js')`
3. 部署包中 `native/` 目录放置于 `resources/native/`（匹配 electron-builder `extraResources` 规范）

**代码变更**：
```typescript
// 修复前
nativeModule = require(path.join(__dirname, '../native/index.js'));

// 修复后
const nativePath = process.resourcesPath
  ? path.join(process.resourcesPath, 'native/index.js')
  : path.join(__dirname, '../../../native/index.js');
nativeModule = require(nativePath);
```

**问题描述**：  
运行解压后的便携版时，Electron 主进程立即崩溃，报错：
```
ReferenceError: exports is not defined in ES module scope
This file is being treated as an ES module because it has a '.js' file
extension and 'package.json' contains "type": "module".
```

**根本原因**：  
`package.json` 中 `"type": "module"` 使 Node.js 以 ESM 模式加载所有 `.js` 文件。`dist-electron/` 中的 Electron 代码由 tsc 编译为 CommonJS 格式（使用 `require()` 和 `exports`），两者冲突。

**修复方案**：
1. 新增 `scripts/rename-electron-output.cjs` 后处理脚本，在 tsc 编译后将 `.js` 重命名为 `.cjs`
2. `package.json` 的 `main` 字段更新为 `dist-electron/main.cjs`
3. `electron/main.ts` 中 preload 引用更新为 `preload.cjs`
4. `electron:ts` 脚本追加 `&& node scripts/rename-electron-output.cjs`
5. `electron/preload.d.ts` 添加 `getPathForFile` 类型定义（之前遗漏）

- **前端**：`DragOverlay.tsx`、`FileChipBar.tsx`、`ipc.ts`、`App.tsx`
- **Electron 主进程**：`preload.ts`、`main.ts`
- **原生模块**：无变更（仅版本号）
- **向后兼容**：完全兼容。`getFilePath` 的三层降级确保在不支持 `webUtils.getPathForFile` 的旧版 Electron 中自动回退到 `File.path`。
