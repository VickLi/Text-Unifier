# 产品需求文档（PRD）极简版 V3.2：文档终版确定器 Text Unifier

| 项目名称 | 文档终版确定器（Text Unifier） |
| :--- | :--- |
| **版本号** | V3.2 |
| **版本类型** | 功能增强 + UI 重构 + 新增对比模式 |
| **目标平台** | Windows / macOS 桌面端（Electron + napi-rs） |
| **基线版本** | V3.1（含小说文本清洗引擎） |

---

## 版本概述

V3.2 聚焦 **三大方向**：① 界面布局重设计（文件区轻量化、预览区主位化）；② 预览编辑与多步撤回；③ 新增文档对比模式。共 5 项需求。

| 编号 | 需求 | 类型 | 用户痛点 |
| :---: | :--- | :--- | :--- |
| RQ-01 | 文件列表收缩为浮动标签栏 | UI 重构 | 文件区太大挤占去重结果空间 |
| RQ-02 | 上传区改为按钮 + 全窗口拖拽遮罩 | UI 重构 | 拖拽上传区占位过大，且拖拽反馈不明显 |
| RQ-03 | 预览区可编辑 + 5 步撤回栈 | 新功能 | 清洗/排版操作无法预览效果、不能撤销 |
| RQ-04 | 预览区主位化 + 悬停溯源 | UI 增强 | 预览空间不足；需确认段落来源 |
| RQ-05 | 文档对比模式 | 全新功能 | 需要比对两个文档的异同 |

## 1. 功能边界

### 1.1 做什么

#### RQ-01：文件列表收缩为浮动标签栏

- **替换**当前"白色卡片 + 垂直列表"的文件展示方式
- 文件以**横向滚动芯片/标签（Chips）**形式展示在拖拽区下方
- 每个芯片显示：文件图标 + 文件名（截断至 20 字） + 大小标签 + 删除 × 按钮
- 第 1 个芯片有 **★ 主文件** 徽章
- 拖拽芯片可**横向排序**（替代原来的垂直拖拽排序）
- 芯片行高度 ~36px，远小于原来的 44px × N 的高度占用
- 芯片行右侧始终显示「+ 添加文件」按钮

#### RQ-02：上传区改为按钮 + 全窗口拖拽遮罩

- **移除**当前大面积虚线拖拽框
- 改为一个 **紧凑按钮**：「📁 添加 .txt 文件」（置于文件芯片行内或标题栏旁）
- **拖拽行为变更**：
  - 用户从操作系统拖动文件到软件窗口时
  - 整个窗口覆上一层 **半透明蓝色遮罩（DragOverlay）**，居中显示：
    - 📁 大图标
    - "释放到此添加 TXT 文件"
    - "仅支持 .txt 格式，可多选"
  - 文件拖出窗口 → 遮罩消失
  - 松开鼠标 → 遮罩消失，触发文件解析
- DragOverlay 在 `document` 级别监听 `dragenter`/`dragleave`（而非局部组件）

#### RQ-03：预览区可编辑 + 5 步撤回栈（UndoStack）

- **预览区变为可编辑文本框**（`contentEditable` 或 `textarea`），用户可直接修改合并后的文本
- **操作历史栈**：所有导致预览文本变化的操作自动入栈
  - 清洗/章节/排版操作
  - 用户手动编辑
  - 重复组勾选（改变预览内容）
- **栈容量**：5 步（FIFO，满 5 步后最早快照出栈）
- **撤回操作**：
  - `Ctrl+Z` → 撤销到上一步快照
  - `Ctrl+Y` / `Ctrl+Shift+Z` → 重做到下一步
  - 界面提供「↶ 撤回」「↷ 重做」按钮
- **快照内容**：保存当前 `previewParagraphs` 数组 + `paragraphCheckedMap` 的深拷贝
- **栈重置时机**：用户拖入新文件/重新开始 → 清空撤回栈

#### RQ-04：预览区主位化 + 悬停溯源

- **预览区**在水平空间上占比 ≥ 55%
- **重复组列表**宽度从 380px 收缩至 **280px**（可拖拽调整）
- 鼠标悬停预览段落时，显示 **Tooltip 浮层**：
  - 格式：`来源：文件A.txt (第 12 段), 文件B.txt (第 5 段)`
  - 若段落仅来自 1 个文件：`来源：文件A.txt`
  - 若段落来自全部文件：`来源：全部文件 (N 个)`
- Tooltip 位置跟随鼠标，带 8px 偏移

#### RQ-05：文档对比模式

- **双模式切换**：标题栏下方增加 Tab 切换条
  - Tab「🔗 合并去重」→ 现有功能
  - Tab「📋 文档对比」→ 新功能
- **对比模式交互**：
  - 用户导入 **恰好 2 个** TXT 文件（多于 2 个提示"对比模式仅支持 2 个文件"）
  - 两个文件分别显示在左栏和右栏
  - 对比前复用归一化管线（统一换行/空格）
  - 段落对齐：以 `\n\n` 为段落边界，LCS（最长公共子序列）算法对齐相同段落
  - 视觉效果：
    - 绿色背景 → 两个文件内容相同的段落
    - 红色背景 → 仅在左栏出现的段落（左栏独有）
    - 蓝色背景 → 仅在右栏出现的段落（右栏独有）
    - 灰色背景 → 两个文件都有但内容有差异的段落（逐词 diff 高亮）
  - 中间联动滚动：左右栏同步垂直滚动
  - 统计栏：共有段落 N 段 / 左独有 M 段 / 右独有 K 段

### 1.2 不做什么

- 文档对比不支持 3 个及以上文件（V3.2 范围外）
- 对比模式不输出合并结果（仅审阅，不导出合并文档）
- 浮动标签栏不支持换行（超出宽度横向滚动）
- 预览编辑不提供富文本（纯文本编辑）
- 撤回栈不持久化（刷新/重启后清空）

## 2. 核心业务逻辑

### 2.1 文档对比 — 段落对齐算法

```text
输入: FileA (归一化), FileB (归一化)
        ↓
Step 1: 按 \n\n 分割为段落数组 PA[], PB[]
        ↓
Step 2: LCS 最长公共子序列 → 对齐表 Alignment[]
        │
        │  Alignment[i] = {
        │    type: 'match'     → PA[i] == PB[j], 绿色
        │    type: 'leftOnly'  → PA[i] 独有, 红色
        │    type: 'rightOnly' → PB[j] 独有, 蓝色
        │    type: 'diff'      → PA[i] ≈ PB[j] (内容相似但不同), 灰色 + 逐词标红
        │  }
        ↓
Step 3: 渲染双栏同步滚动视图
```

**相似度判定**：
- 若两个段落的内容相似度 > 60%（基于 Levenshtein 距离），归类为 `diff`
- 否则归类为各自独有的段落

### 2.2 撤回栈（UndoStack）

```text
class UndoStack(maxDepth = 5):
    stack: Snapshots[]
    pointer: int  // 当前快照位置

    push(snapshot):
        if pointer < stack.length - 1:
            stack = stack[0..pointer+1]  // 新操作丢弃"未来"的重做
        stack.push(snapshot)
        if stack.length > maxDepth:
            stack.shift()  // 丢弃最早快照
        pointer = stack.length - 1

    undo():
        if pointer <= 0: return stack[0]
        pointer -= 1
        return stack[pointer]

    redo():
        if pointer >= stack.length - 1: return stack[-1]
        pointer += 1
        return stack[pointer]

    clear():
        stack = []
        pointer = -1
```

**快照内容**：
```typescript
interface Snapshot {
    paragraphs: PreviewParagraph[];      // 深拷贝
    checkedMap: Record<string, boolean>; // 深拷贝
    timestamp: number;                   // 用于 UI 展示"X秒前的操作"
}
```

**触发快照保存的操作**：
| 操作 | 触发时机 |
| :--- | :--- |
| 点击「应用处理」 | 排版完成后 |
| 用户手动编辑预览区 | 输入停止 500ms 后（debounce） |
| 勾选/取消勾选段落 | 每次 Checklist 变更 |

### 2.3 文件标签栏排序

```text
FileChips = [F1★, F2, F3, ...]
              ↓ 用户拖拽 F3 到 F1 前
FileChips = [F3★, F1, F2, ...]
              ↓ 触发重新分析（F3 成为主文件）
```

横向拖拽使用 `@dnd-kit` 的 `horizontalListSortingStrategy`。
